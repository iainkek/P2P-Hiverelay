# v0.7 Known Limitations & Follow-Up Work

Status: **open**. Created when v0.7.0 shipped (anchor status, repair loop,
follow-anchored, signed proofs). These are the gaps I want addressed before
we tell the community the network is "production-grade resilient" without
qualifications.

Each item lists the current state, the failure it permits, severity, and
the concrete fix.

---

## 1. "Anchored" is binary, not granular

**Current behavior**
`appRegistry.setAnchored(appKey, length)` flips `anchored: true` as soon as
`drive.version > 0`. A drive with only metadata replicated (1 block of
~few kB) is reported as anchored alongside a fully-replicated drive
(thousands of blocks).

**What this permits**
A relay's catalog says `anchored: 53` even if it only has metadata for
all 53 — clients reading via the gateway will hang waiting for content
blocks that the relay doesn't actually hold.

**Severity** — moderate. Misleading rather than malicious. Repair loop
keeps trying to download blob content; eventually catches up. But the
honesty guarantee we make to clients is weaker than the bool implies.

**Fix**
Track `bytesStored / totalBytes` per drive and expose as `anchorPct`. Mark
`anchored: true` only above a configurable threshold (default 100% or
≥95% of bytes). Surface in `/api/anchors` and the signed proof.

```js
// Sketch
node.appRegistry.setAnchored(key, {
  metadataLength: drive.version,
  blobBytes: drive.blobs?.contentBlocksDownloaded,
  totalBytes: drive.blobs?.totalContentBlocks
})
```

## 2. Follow-anchored is a storage attack vector

**Current behavior**
With `followAnchoredFromPeers: true`, any peer can claim `anchored: true`
in its catalog broadcast and our relay auto-seeds the drive — opening a
session, joining the swarm, attempting downloads forever.

**What this permits**
A malicious or compromised peer floods our catalog handler with bogus
drive keys. Our relay creates Hyperdrive instances + Corestore entries
for each, joins discovery topics, and runs the repair loop trying to
download non-existent content. CPU + memory + DHT pressure scale with
the attacker's claims.

**Severity** — high if `followAnchoredFromPeers` is on. We mitigated with:
- Storage ceiling at 80%
- Reputation gate (default 0, operator-tightenable)
- Per-peer 30s throttle + 10 apps/event cap

But none of these prove the peer actually has the data. The attack still
works, just slower.

**Default state today** — off. Marked clearly in CHANGELOG so operators
who turn it on understand the risk.

**Fix**
Verify-before-follow:
1. When peer A's catalog says `anchored: true` for drive K
2. Before our relay accepts and seeds K, fetch
   `https://A/api/anchors/K/proof`
3. Verify signature against A's identity pubkey
4. Only then `seedApp(K, ...)`

If A can't produce a valid proof in N seconds, treat the catalog claim
as suspect and don't follow. The proof endpoint already exists — this
is wiring, not a new primitive.

```js
// Sketch in the catalog handler
if (followThisAnchored) {
  const proof = await this._fetchPeerProof(peerKey, appKey)
  if (!proof || !proof.verified) {
    this.emit('catalog-sync-rejected', { appKey, reason: 'unverifiable proof' })
    continue
  }
  // …then seedApp
}
```

## 3. Signed proof is "claim of state," not "proof of possession"

**Current behavior**
`/api/anchors/<key>/proof` returns the relay's identity-key-signed claim
that it has anchored the drive. The signature proves the *relay made the
claim*, not that the relay actually holds blocks.

**What this permits**
A malicious relay can sign whatever proof it wants. Currently no way for
a verifier to detect this from the proof alone — they'd have to attempt
a content fetch via the gateway and see if it actually returns bytes.

**Severity** — moderate. The verifier already detects content-divergence
between relays via `compareDrive()`, which catches lying-about-content.
But anchor-state lies aren't caught by content-comparison if the lying
relay simply refuses to serve.

**Fix — block-of-possession challenge**
Extend the proof endpoint with a challenge mode:
```
POST /api/anchors/<key>/proof
Body: { challenge: <random 32-byte hex>, blockIndex: <integer> }
```

Relay reads block at `blockIndex`, hashes (challenge || block), signs
(blockIndex || hash) with identity key. Verifier checks:
- Signature valid for relay pubkey
- Hash matches what verifier independently computed by fetching the
  block via the gateway

A relay that doesn't have the block at `blockIndex` cannot respond
correctly. This makes the proof unforgeable.

The proof-of-relay protocol already exists for circuit-relay challenges;
this is a parallel for content possession.

## 4. Cold-start primer trusts the seed list blindly

**Current behavior**
`config.coldStartRelays = ['https://A', 'https://B']` → fresh relay
fetches catalogs from A and B over HTTPS and follows their anchored
entries (subject to follow-anchored gating).

**What this permits**
If A or B is malicious, the fresh relay starts polluted from boot. Same
attack as #2, scoped to the cold-start window.

**Severity** — moderate. Limited to operator-chosen URLs, which should
be trusted (e.g. foundation relays). Not exploitable by random network
peers.

**Fix**
Apply the same verify-before-follow logic from #2 to cold-start
entries. Each anchored claim from a primer URL must produce a valid
signed proof before we'll seedApp it.

Optionally: pin specific relay pubkeys in cold-start config and require
the served capability doc's pubkey to match.

```js
config.coldStartRelays: [
  { url: 'https://relay-us.p2phiverelay.xyz', pubkey: '37cf4bfb...' }
]
```

## 5. No deletion / unseed propagation beyond direct hops

**Current behavior**
`unseedApp` with a signed unseed message removes the drive from our
relay and broadcasts to **directly-connected** peers. Relays not
connected at the moment of broadcast never see it. Relays that
acquired the drive via follow-anchored never see the original
publisher's unseed because they're peered with another relay, not the
publisher.

**What this permits**
- Author signs unseed — their drive disappears from relays they're
  directly peered with
- Same drive lives on other relays that pulled it via cross-relay
  replication, with no way for the author to reach them

**Severity** — depends on perspective. **Feature** for
censorship-resistance. **Bug** for operators who promised users they
could "kill switch" their content.

**Fix options**
- **Gossip-the-unseed**: relays that receive an unseed re-broadcast it
  to their own peers, bounded by hop count + dedup nonce. Spreads in
  O(network diameter) hops.
- **Signed-revocation list**: maintain a published list of unseed
  signatures that all relays poll periodically. New relays catching up
  see "this drive was unseeded by author X at time T" and refuse to
  serve.
- **Document as not-a-bug**: be clear in the threat model that
  censorship-resistance and unseed-propagation are in tension; if
  you want hard delete, don't seed to relays you don't control.

I'd default to gossip-the-unseed because it preserves
censorship-resistance for content the author *didn't* unseed.

## 6. Scaling pressure: quadratic peer load

**Current behavior**
Each periodic repair pass (every 10 min) iterates every unanchored
entry and tries to pull blocks from any peer with the discovery key.
At N relays × M drives, that's roughly O(N × M) repair attempts per
cycle.

**What this permits**
At 100 relays × 1000 drives, ~100k coordination events per 10-min
cycle. DHT load grows. Relay CPU climbs.

**Severity** — low today (we're at 5 relays × 50 drives). Becomes
real around 50 relays × 500 drives.

**Fix**
- Region-sharded discovery topics (already in code, off by default —
  `regionTopic(code)` helper exists, just not auto-joined yet)
- Repair-budget per cycle (don't try to repair every drive in one pass)
- Adaptive backoff (drives that have been unanchored for hours get
  exponentially-larger retry intervals)

## 7. Storage growth is now operator-determined, not user-determined

**Current behavior**
Pre-v0.7: relay storage = sum of drives users seeded to it.
With v0.7 + `followAnchoredFromPeers: true`: relay storage grows
toward `maxStorageBytes × followAnchoredStorageCeiling` (default 80%)
because the relay actively follows anchored content from peers.

**What this permits**
A new operator who doesn't read the docs turns on follow-anchored
expecting "be a good citizen" semantics, then discovers their disk
fills up to 80% within hours/days.

**Severity** — low. Documentation issue + operator-onboarding wizard
should make this clear at setup time.

**Fix**
- Wizard: ask explicitly "do you want to mirror network content, or
  only host what users seed to you?" and set follow-anchored
  accordingly
- Dashboard: add a "Storage forecast" panel showing growth rate +
  projected days-until-full
- Metric: emit `storage-growth-rate` events for alerting

## 8. Privacy: anchored claims leak hosting metadata

**Current behavior**
Capability doc + catalog tell the world exactly which drives the relay
has anchored. For non-blind drives this isn't new (the registry was
already public). For blind drives, the *fact* of hosting was known but
the anchor status wasn't.

**What this permits**
An adversary correlating "I have the encryption key for blind drive K"
with "this relay anchored K" can prove the relay was the one serving
the encrypted bytes during a specific time window.

**Severity** — low to moderate, depending on use case. Blind drive
operators in adversarial environments need to know.

**Fix**
- Optional: relay config flag `redactAnchorState: true` — capability
  doc says total but not anchored counts; signed proofs require
  authenticated request from the drive's author.
- Document the leak in the threat model so users with
  privacy-sensitive use cases choose appropriate relays.

## Priority order for the next milestone

If we tackle a v0.7.1 follow-up release:

1. **#2 fix (verify-before-follow)** — closes the main attack vector
2. **#3 fix (block-of-possession proof)** — closes the lying-relay
   attack vector and makes the network trust model load-bearing
3. **#1 fix (granular anchored)** — improves honesty without
   security-changing implications
4. **#5 fix (gossip-the-unseed)** — depending on community feedback;
   could be a feature flag
5. **#6 fix (region-sharded topics)** — preemptive scaling work
6. **#4, #7, #8** — config + documentation hygiene; low effort

## Testing strategy

For #2 and #3 specifically — both involve cryptographic proofs and
adversarial scenarios. Tests should include:
- Honest-relay path (proof verifies, drive accepted)
- Lying-claim path (relay claims anchored but no signed proof — rejected)
- Stale-proof path (proof is older than acceptable window — rejected)
- Pubkey-mismatch path (signature is valid but for wrong key — rejected)
- Block-of-possession (relay can produce hash for owned blocks; cannot
  for unowned blocks)

These belong in `test/unit/anchor-proof.test.js` extensions plus a new
`test/integration/anchor-trust.test.js` for the end-to-end flow.
